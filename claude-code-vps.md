
# Работа с Claude Code на VPS

Инструкция для разработчика. На сервере проекта работает Claude Code; общаться с ним можно из любого места и из любого инструмента. **Основной способ — HTTP API** (раздел 2). Для работы из терминала есть SSH-клиент `cc` (раздел 3). Оба способа — двери к одним и тем же чатам.

> Токен доступа к API выдаёт администратор — в этом документе он не публикуется.

---

## 1. Как это устроено

- **Чат = имя.** `report`, `ivan-parser` — независимые диалоги, каждый со своей историей. Контекст живёт на сервере: продолжать чат можно с любой машины, из API и из терминала вперемешку. Не хотите придумывать имя — начните чат с `@`, имя сгенерирует сам Claude по первому вопросу.
- **У каждого чата рабочая папка** на сервере. Файлы, которые вы загружаете, попадают в неё; файлы, которые Claude создаёт по заданию, появляются там же и возвращаются в ответе.
- **Диалог двусторонний.** Claude настроен задавать уточняющие вопросы, когда данных не хватает, а не молча угадывать. Вопрос приходит обычным ответом — просто отправьте следующий ход в тот же чат.
- **Долгие задачи** запускаются фоном: поставили, отключились, забрали результат позже.
- **Один ход в чате за раз**: параллельные обращения к одному чату выстраиваются в очередь, разные чаты работают параллельно.

## 2. HTTP API — основной способ

| Параметр | Значение |
|---|---|
| Адрес | `http://72.56.82.239:8377` |
| Аутентификация | заголовок `Authorization: Bearer <токен>` |
| Токен | выдаёт администратор (подставляйте вместо `$TOKEN`) |
| Кодировка тела | всегда UTF-8 |

### Эндпоинты

| Метод и путь | Что делает |
|---|---|
| `POST /ask` | Новый чат (имя придумает Claude), тело = промпт → ответ-конверт |
| `POST /ask/<чат>` | Ход в чате (чат создастся при первом обращении) |
| `POST /bg/<чат\|@>` | Фоновая задача → `{"job": "job-..."}` |
| `GET /job/<job-id>` | Статус фоновой задачи; когда `done` — сразу с результатом |
| `GET /chats` | Список чатов |
| `GET /files/<чат>` | Файлы в папке чата |
| `GET /file/<чат>/<имя>` | Скачать файл |
| `PUT /file/<чат>/<имя>` | Загрузить файл (тело = содержимое) |

### Быстрый старт

```bash
B=http://72.56.82.239:8377
H="Authorization: Bearer $TOKEN"   # токен — у администратора

# новый чат — имя придумает Claude (придёт в поле chat)
curl -H "$H" --data-binary "Помоги спланировать регламент бэкапов MySQL" $B/ask

# продолжение диалога — по имени из ответа
curl -H "$H" --data-binary "Разверни пункт 2 подробнее" $B/ask/mysql-backup-plan
```

### Формат ответа

Каждый `POST /ask...` возвращает один и тот же конверт — вашему коду не нужно заранее знать, чем закончится ход (текстом, файлами или встречным вопросом):

```json
{
  "chat": "report",
  "answer": "Готово: сводка в summary.json. Уточни: включать ли декабрь?",
  "files": [
    { "name": "summary.json", "size": 412, "content": "{ ... }" },
    { "name": "big-dump.bin", "size": 10485760 }
  ],
  "meta": {
    "session_id": "...", "num_turns": 3, "duration_ms": 8525,
    "total_cost_usd": 0.0994, "is_error": false
  }
}
```

- `files` — файлы, созданные/изменённые Claude **за этот ход** (не вся папка). Текстовые до 256 КБ приходят сразу с `content`; крупные и бинарные — только имя и размер, забирайте через `GET /file/...`.
- Если в `answer` встречный вопрос — следующий `POST /ask/<чат>` с ответом продолжит работу.
- Нужен машиночитаемый ответ — попросите в промпте: «ответь строго JSON без пояснений», и парсите `answer`.

### Файлы в обе стороны

```bash
# загрузить файл в чат и спросить по нему
curl -H "$H" -X PUT --data-binary @orders.json $B/file/report/orders.json
curl -H "$H" --data-binary "Посчитай сумму по orders.json, сохрани в sum.json" $B/ask/report

# забрать результат
curl -H "$H" $B/file/report/sum.json -o sum.json
```

### Долгие задачи

У `/ask` таймаут 30 минут, и HTTP-соединение должно жить всё это время. Для долгих задач — фоновый режим:

```bash
JOB=$(curl -H "$H" --data-binary "Разбери все CSV и собери сводку" $B/bg/report | jq -r .job)
curl -H "$H" $B/job/$JOB     # {"status": "running"} ... затем {"status": "done", "result": {конверт}}
```

Результаты фоновых задач хранятся 7 дней.

### Пример на Python

```python
import requests

B = "http://72.56.82.239:8377"
H = {"Authorization": "Bearer " + TOKEN}

r = requests.post(f"{B}/ask/report",
                  data="Что в sum.json?".encode("utf-8"),
                  headers=H, timeout=600).json()
print(r["answer"])
for f in r["files"]:
    print("файл:", f["name"], f.get("content", f"({f['size']} байт — скачать отдельно)"))
```

### Замечания

- API работает по HTTP без шифрования: не используйте из недоверенных сетей, токен не публикуйте. Утёк токен — сообщите администратору, он заменит его за минуту.
- Windows-консоль часто отправляет кириллицу не в UTF-8 — тогда Claude получает кракозябры. Надёжный способ: промпт файлом `curl --data-binary @prompt.txt` (файл в UTF-8).

## 3. Терминал — SSH-клиент `cc` (альтернатива)

Для тех, кто работает из консоли: те же чаты, плюс живой REPL-режим с потоковым выводом.

### Доступ по ключу

1. Сгенерируйте ключ: `ssh-keygen -t ed25519` (Enter на все вопросы). Публичную часть (`cat ~/.ssh/id_ed25519.pub`; в PowerShell — `type $env:USERPROFILE\.ssh\id_ed25519.pub`) отправьте администратору — она не секрет. Приватный файл никому не передавайте.
2. Администратор добавляет её в `/root/.ssh/authorized_keys` одной строкой с ограничением (без префикса ключ даст полный root!):
   ```text
   command="/usr/local/bin/cc-shell",restrict ssh-ed25519 AAAA... ivan@laptop
   ```
   Такой ключ позволяет выполнять **только** `cc-ask`: ни shell, ни scp, ни проброса портов. Отзыв доступа — удаление строки.
3. Проверка: `ssh root@147.45.137.111 'cc-ask --list'`.
4. Установите клиент (раздаётся с самого сервера):
   ```bash
   mkdir -p ~/bin && ssh root@147.45.137.111 'cc-ask --client' > ~/bin/cc && chmod +x ~/bin/cc
   ```
   Если доступа ещё нет — файл `cc` можно получить у администратора и начать с `cc setup`: он сам сгенерирует ключ, покажет строку для администратора и пропишет алиас в `~/.ssh/config`.

### Команды

Промпт передаётся через stdin (основной способ — любые символы и длина) или аргументом (короткие вопросы; спецсимволы `; | & > <` в аргументе блокируются защитой ключа).

```bash
echo "промпт" | cc <чат>           # ход диалога (ответ + список новых файлов)
cc @ "новая задача"                 # новый чат, имя придумает Claude: [чат: <имя>]
cc chat <чат|@>                     # живой диалог (REPL, ответы потоком)
echo "промпт" | cc --json <чат>     # ответ JSON-конвертом (как в API)
echo "промпт" | cc --stream <чат>   # ответ потоком по мере генерации
echo "промпт" | cc --bg <чат|@>     # фоновая задача -> job-id
cc --status job-... / --result job-... / --jobs
cc --send <чат> <файл> "промпт" < файл    # файл + вопрос одним вызовом
cc --put <чат> <файл> < файл        # просто загрузить файл
cc --untar <чат> < архив.tgz        # загрузить несколько файлов
cc --files <чат> / --get <чат> <файл> / --tar <чат> > все.tgz
cc --list / --new <чат>             # список чатов / сброс контекста
cc --system <чат> "текст"           # системный промпт чата ("" — убрать)
cat task.md | cc <чат>              # длинное задание из файла
```

### Живой диалог (REPL)

```text
$ cc chat @
вы> Помоги выбрать формат логов для сервиса
[чат: service-log-format]
Пара уточняющих вопросов: какой объём логов в сутки и куда складываете?
вы> Около 2 ГБ, в Elasticsearch
Тогда рекомендую структурированный JSON... (текст появляется по мере генерации)
вы> /put sample.log        # загрузить файл из диалога
вы> /get report.json       # скачать результат
вы> exit
```

## 4. Ограничения и правила

- **Параллельность.** Сервер спокойно тянет 5–8 одновременных запросов. Массовые батчи согласуйте: на сервере работает продакшн-сайт, а лимиты аккаунта Claude общие на всех.
- **Хранимых чатов — сколько угодно**, неактивный чат ничего не потребляет. Но не складируйте в папки чатов гигабайты — следите за диском.
- **Один чат — одна задача.** Длинные «вечные» чаты дорожают (в контекст уходит вся история, см. `total_cost_usd` в `meta`) и упираются в лимит контекста. Закончили тему — новый чат или сброс (`--new` / попросите администратора).
- Системный промпт чата настраивается через терминальный канал (`cc --system`); в API-чатах он тоже действует.

## 5. Если что-то не работает

| Симптом | Причина и решение |
|---|---|
| API: `{"error": "unauthorized"}` | Нет/неверный заголовок `Authorization: Bearer <токен>` |
| API: Claude отвечает невпопад на кириллицу | Тело ушло не в UTF-8 (частая проблема Windows-консоли). Промпт файлом: `--data-binary @prompt.txt` (UTF-8) |
| API: 504 или обрыв на долгой задаче | Используйте `POST /bg/<чат>` + `GET /job/<id>` |
| SSH: сервер спрашивает пароль | Ключ не добавлен администратором или ssh берёт не тот: `ssh -i ~/.ssh/id_ed25519 ...` |
| SSH: `Permission denied (publickey)` | Ключ не добавлен или строка в `authorized_keys` повреждена |
| SSH: ключ отклоняет команду со спецсимволами | Защита ограниченного ключа: промпт с `; \| & > <` передавайте через stdin |
| PuTTY/plink молча зависает | Известная проблема с этим сервером — используйте OpenSSH (`ssh`) |
| `чат занят другим ходом` | В чат одновременно пишет кто-то ещё; повторите позже или используйте свой чат |
| Ответ «не помню контекст» | Опечатка в имени чата (`GET /chats` / `cc --list`) — новое имя = новый пустой чат |
