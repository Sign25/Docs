## 8.1 Назначение документа

Модуль ADOLF CFO в текущей версии формирует P&L-отчёты и ABC-анализ на основе данных маркетплейсов (Wildberries, Ozon, Яндекс.Маркет). Параллельно в PostgreSQL уже поступают данные из 1С:Комплексная автоматизация 2 — 19 регулярных выгрузок (запросы Q-01…Q-18), охватывающих бухгалтерские регистры, остатки, заказы, документы и справочники.

Документ определяет:

- какие существующие отчёты и эндпоинты модуля CFO обогащаются данными 1С (§8.4);
- какие новые отчёты и эндпоинты становятся возможными (§8.5);
- технические требования к интеграции (§8.6);
- приоритеты внедрения (§8.7);
- структуры используемых таблиц PostgreSQL (§8.8, Приложение А).

## 8.2 Доступные данные 1С в PostgreSQL

Сводка выгрузок по группам (полные структуры — в Приложении А и документе интегратора «1C_Export»):

| Группа | Таблицы PostgreSQL | Содержание |
|--------|--------------------|------------|
| Бухгалтерские регистры | `1C_account_turns_90`, `1C_account_turns_90_expenses`, `1C_account_balance_41`, `1C_account_turns_cash`, `1C_account_turns_settlements` | Выручка и себестоимость (сч. 90.01.1 / 90.02.1), коммерческие и управленческие расходы, стоимость запасов (сч. 41), денежные средства (сч. 50/51), расчёты с контрагентами (сч. 60/62) |
| Остатки и заказы | `1C_stock_balance`, `1C_customer_orders_*`, `1C_supplier_orders_*`, `1C_goods_receipts_*` | Ежедневные остатки по складам, открытые заказы клиентов и поставщикам, поступления товаров |
| Справочники | `1C_nomenclature`, `1C_characteristic`, `1C_counterparties`, `1C_contracts` | Номенклатура (вид = бренд), характеристики со штрихкодами, контрагенты, договоры |
| Цены | `1C_nomenclature_prices`, `1C_supplier_prices` | Срезы последних цен номенклатуры и поставщиков |
| Документы | `1C_sales_*`, `1C_returns_*`, `1C_inventory_audit_*` | Реализации, возвраты с причинами, инвентаризации |
| Обороты | `1C_stock_turnover` | Приход/расход по складам в штуках и деньгах |

**Ключ сопоставления с данными маркетплейсов:** поле `article` (артикул = SKU = offer_id). Присутствует во всех товарных таблицах. Резервный ключ — штрихкод EAN-13 из `1C_characteristic`.

## 8.3 Текущий API модуля (CFO REST API v0.1.1)

Базовый путь `/api/v1/cfo`. Эндпоинты на момент написания:

| Метод | Путь | Назначение |
|:-----:|------|------------|
| GET | `/health` | Liveness-проверка |
| GET | `/api/v1/cfo/pnl/category` (+ `/export`) | P&L по категориям (WB) |
| GET | `/api/v1/cfo/pnl/marketplace` (+ `/export`) | P&L по маркетплейсам |
| GET | `/api/v1/cfo/pnl/sku` | P&L по SKU (фильтры + пагинация) |
| GET | `/api/v1/cfo/abc` (+ `/export`) | ABC-анализ SKU по марже (WB) |
| GET/POST | `/api/v1/cfo/exclusions` | Глобальные исключения |
| GET | `/api/v1/cfo/categories` | Справочник родитель → подкатегории |

Конвенции, которым должны следовать и новые эндпоинты (§8.5):

- период — `preset` (`yesterday | week | month | year`) либо `from`/`to`, дефолт `yesterday`, half-open границы дня;
- денежные значения — `float`, рубли; проценты — пункты 0…100;
- пагинация `limit`/`offset` с блоком `pagination` и `summary` по полному результату;
- ошибки — `{"detail": "..."}`, HTTP 422;
- глобальные исключения применяются ко всем расчётам;
- кеш на уровне SQL с TTL `CFO_API_CACHE_TTL`; загрузка новых данных 1С должна учитываться при инвалидации (см. §8.6).

## 8.4 Обогащение существующих отчётов

### 8.4.1 Себестоимость (COGS) в P&L и ABC-анализе — критично

**Источник:** `1C_account_turns_90` (Q-01, структура — А.1), фильтр `account_dt = '90.02.1'`, агрегация по `article` за период.

```sql
-- VIEW cfo_v_cogs: фактическая себестоимость продаж по SKU за месяц
SELECT period, article,
       SUM(amount_dt)   AS cogs,
       SUM(quantity_dt) AS qty_sold
FROM "1C_account_turns_90"
WHERE account_dt = '90.02.1'
GROUP BY period, article;
```

Сейчас поля `cogs`, `gross_profit`, `net_profit` во всех эндпоинтах `/pnl/*` рассчитываются без фактической себестоимости из учёта. Подключение Q-01 делает достоверными все три P&L-отчёта и переводит ABC-анализ с классификации «по выручке» на классификацию «по фактической марже». Без этого шага класс D (убыточные SKU) определяется неточно.

**API:** новые эндпоинты не требуются. Затрагиваются ответы `GET /pnl/category`, `GET /pnl/marketplace`, `GET /pnl/sku`, `GET /abc` и их `/export`: поля `cogs`, `gross_profit`, `net_profit`, `gross_margin_pct`, `net_margin_pct` начинают рассчитываться от фактического COGS.

### 8.4.2 P&L по брендам и обогащение справочника категорий

**Источники:** `1C_nomenclature` (Q-10, А.9) — поле `kind` содержит бренд (Охана Маркет / Охана Кидс), `parent` — группу; `1C_characteristic` (Q-10/1, А.10) — размеры/цвета и штрихкоды.

Отчёт «P&L по брендам» заявлен в матрице доступа (Senior/Director), но на данных маркетплейсов его собрать не из чего. Справочник 1С даёт устойчивую привязку SKU → бренд → группа и сокращает долю строк «Прочее» в `/pnl/category`. Дополнительно появляются разрезы: сезон, размерный ряд (через характеристики).

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/pnl/brand` (+ `/export`) | `preset` / `from`+`to`, `brand` (optional, exact match по `kind`) | Модель `PnLByCategoryReport` с полем `brand` вместо `category`; строки — бренды, при фильтре `brand` — drill-down по категориям внутри бренда |

Затрагиваются также `GET /categories` (источник иерархии — `1C_nomenclature.parent`) и качество группировки в `GET /pnl/category`.

### 8.4.3 Причины возвратов в алертах и анализе убыточности

**Источник:** `1C_returns_header` / `1C_returns_lines` (Q-15, А.11) — поле `reason` в шапке документа.

Маркетплейсы дают факт возврата и стоимость обратной логистики, но не причину. Данные 1С позволяют разделить возвраты «по браку» и «не подошло» и превратить алерт об убыточном SKU в диагноз: проблема качества, размерной сетки или экономики логистики.

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/returns` | `preset` / `from`+`to`, `sku`, `reason` (optional) | Строки: `article`, `return_number`, `return_date`, `marketplace`, `quantity`, `amount`, `reason`; `summary` с разбивкой сумм по причинам |

Затрагиваются алерты об убыточных SKU и AI-инсайты: в payload алерта добавляется топ причин возвратов по SKU.

### 8.4.4 Сверка «1С ↔ маркетплейсы»

**Источники:** `1C_sales_header` / `1C_sales_lines` (Q-14, А.12) против нормализованных транзакций маркетплейсов; дополнительно выручка `90.01.1` из Q-01 в разрезе контрагентов (WB / Ozon / ЯМ).

Регулярная сверка количества и сумм отгрузок выявляет потерянные отгрузки, неучтённые удержания и ошибки маппинга SKU. Одновременно служит контролем качества данных самого модуля CFO.

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/reconciliation` | `preset` / `from`+`to`, `marketplace`, `only_diff` (bool, дефолт true) | Строки по `article` × маркетплейс: `qty_mp`, `qty_1c`, `amount_mp`, `amount_1c`, `diff_qty`, `diff_amount`; `summary` — итоговые расхождения |

## 8.5 Новые отчёты на данных 1С

### 8.5.1 Запасы и неликвиды

**Вопрос директора:** «Сколько денег заморожено в товаре и что распродавать в первую очередь?»

**Источники:** `1C_stock_balance` (А.6, ежедневные остатки), `1C_account_balance_41` (А.3, стоимость запасов на конец месяца), `1C_stock_turnover` (А.15, движение за 7 дней).

Состав отчёта: остатки в штуках и деньгах по складам и брендам; SKU без движения 30/60/90 дней; оборачиваемость в днях. Ключевая витрина — матрица «ABC-класс по марже × оборачиваемость»: маржинальные, но медленные SKU (кандидаты на пересмотр закупки) и немаржинальные быстрые (кандидаты на повышение цены или вывод).

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/inventory` | `date` (дефолт вчера), `warehouse`, `brand`, `limit`/`offset` | Строки по `article`: `quantity`, `cost_value` (из сч. 41), `warehouse`, `days_no_movement`, `turnover_days`; `summary` — итого стоимость, доля неликвидов |
| GET | `/api/v1/cfo/inventory/abc-turnover` | `preset` / `from`+`to` | Матрица: ABC-класс × сегмент оборачиваемости, в ячейках — число SKU и стоимость остатка |

### 8.5.2 Покрытие запасов и товар в пути

**Вопрос директора:** «На сколько дней хватит остатков по SKU класса A и когда придёт пополнение?»

**Источники:** `1C_stock_balance` (А.6) + темп продаж из данных маркетплейсов + `1C_supplier_orders_header/lines` (А.7/А.8: товар в пути = `quantity − quantity_received`, дата `expected_date`) + `1C_goods_receipts_*` (фактическая скорость пополнения).

Состав отчёта: дни покрытия по SKU, прогнозная дата out-of-stock, ожидаемые поступления. Прямая защита выручки: out-of-stock по классу A — это упущенные продажи, которые модуль сейчас не видит.

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/inventory/coverage` | `sales_window` (дней для темпа продаж, дефолт 28), `abc_class`, `limit`/`offset` | Строки по `article`: `stock_qty`, `daily_sales_rate`, `days_of_cover`, `oos_date` (прогноз), `in_transit_qty`, `expected_date`; сортировка по `days_of_cover ASC` |

### 8.5.3 Операционный P&L (полный)

**Вопрос директора:** «Какая прибыль после всех накладных расходов, а не только после расходов маркетплейсов?»

**Источники:** валовая прибыль из Q-01 (А.1) + `1C_account_turns_90_expenses` (Q-02, А.2: коммерческие 90.07.1 и управленческие 90.08.1 расходы, помесячно).

Закрывает функцию «Консолидированный P&L» из матрицы доступа директора: выручка − COGS − расходы МП − коммерческие − управленческие = операционная прибыль юрлица, помесячно и по организациям.

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/pnl/operating` (+ `/export`) | `from`+`to` (месячная гранулярность), `organization` | Строки по месяцам: `revenue`, `cogs`, `mp_expenses`, `gross_profit`, `selling_expenses` (90.07.1), `admin_expenses` (90.08.1), `operating_profit`, `margin_pct` |

### 8.5.4 Движение денежных средств (ДДС)

**Вопрос директора:** «Сколько денег пришло и ушло за неделю/месяц?»

**Источник:** `1C_account_turns_cash` (Q-04, А.4: счета 50 и 51, обновление еженедельно).

Базовый ОДДС: приход/расход по кассе и расчётным счетам в динамике. Дополняет P&L (начисление) кассовым взглядом — расхождение между прибылью и деньгами становится видимым.

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/cashflow` | `from`+`to`, `organization` | Строки по периодам и счетам (50/51): `inflow` (amount_debit), `outflow` (amount_credit), `net`; `summary` — итого за период |

### 8.5.5 Дебиторская и кредиторская задолженность

**Вопрос директора:** «Сколько нам должны маркетплейсы и сколько мы должны поставщикам?»

**Источник:** `1C_account_turns_settlements` (Q-05, А.5: сальдо Дт по сч. 62 = дебиторка, сальдо Кт по сч. 60 = кредиторка, в разрезе контрагентов).

Особо ценный срез — задолженность маркетплейсов (WB, Ozon, ЯМ как контрагенты по сч. 62): контроль того, что выручка по отчётам МП действительно превращается в выплаты. Динамика месяц к месяцу выявляет рост зависших сумм.

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/settlements` | `from`+`to` (месяцы), `type` (`receivables` / `payables`), `counterparty` | Строки по контрагентам и месяцам: `account`, `counterparty_name` (join `1C_counterparties`), `balance` (Дт для 62 / Кт для 60), `delta_mom` (изменение к прошлому месяцу) |

### 8.5.6 Закупочная аналитика

**Вопрос директора:** «Маржа падает из-за маркетплейса или из-за поставщика?»

**Источники:** `1C_supplier_prices` (Q-17, А.13: прайсы поставщиков), `1C_goods_receipts_lines` (Q-09, А.8: фактические цены приходов), `1C_nomenclature_prices` (Q-13, А.14: собственные типы цен).

Состав: динамика закупочных цен против чистой маржи SKU из модуля CFO; расхождение «прайс vs факт прихода»; сравнение поставщиков по одинаковым артикулам.

**API:**

| Метод | Путь | Параметры | Ответ |
|:-----:|------|-----------|-------|
| GET | `/api/v1/cfo/purchasing/prices` | `sku` или `supplier`, `from`+`to` | Строки по `article`: `supplier_name`, `list_price` (прайс), `receipt_price` (факт прихода), `price_delta_pct`, `net_margin_pct` (из P&L модуля) — тренд по периодам |

## 8.6 Технические требования к интеграции

- Сопоставление выполняется по полю `article`; номенклатура, контрагенты и характеристики приходят GUID-ами и требуют join со справочниками Q-10 / Q-10.1 / Q-11.
- Q-01 выгружается еженедельно со скользящим окном «текущий месяц». Необходимо проверить, сохраняется ли история закрытых месяцев в таблице; если затирается — организовать накопление снапшотов на стороне CFO до подключения COGS.
- Справочники пишутся стратегией TRUNCATE + INSERT: инкрементальная загрузка по `loaded_at` для них неприменима, читать полным срезом.
- Поле `loaded_at` заполняется на стороне PostgreSQL; время серверов 1С и PostgreSQL расходится. Для дедупликации использовать `document_guid` и период, а не время загрузки.
- Заказы поставщикам выгружаются только открытые и только за последние полгода; заказы клиентов — только открытые. Это фильтры на стороне 1С, в расчётах покрытия учитывать.
- Стоимостная оценка в `1C_stock_turnover` построена по виду цены «Цены маркетплейса АКТУАЛЬНЫЕ» — это не себестоимость; для оценки запасов по себестоимости использовать `1C_account_balance_41`.
- Доступ модуля CFO к таблицам `1C_*` организовать через выделенные VIEW (по образцу `cfo_v_cost_prices` из раздела 2 Data Ingestion) и отдельного пользователя БД с правами только на чтение.
- Инвалидация кеша API: расписание выгрузок 1С известно (ежедневные 06:00–07:00, еженедельные Пн 05:00–07:30, ежемесячные 1-е число 05:00–10:00 OMS); после окна загрузки сбрасывать кеш затронутых запросов либо подобрать `CFO_API_CACHE_TTL` так, чтобы данные не отставали более чем на час.

## 8.8 Приложение А. Структуры таблиц PostgreSQL

Структуры приведены по документу интегратора «1C_Export» (запросы Q-01…Q-18). Во всех таблицах `loaded_at TIMESTAMP` заполняется на стороне PostgreSQL.

### А.1 `1C_account_turns_90` (Q-01) — выручка и себестоимость

| Колонка | Тип | Описание |
|---------|-----|----------|
| period | DATE | Период (месяц) |
| nomenclature | VARCHAR(36) | GUID номенклатуры |
| characteristic | VARCHAR(36) | GUID характеристики |
| article | VARCHAR(100) | Артикул (= SKU для маппинга с МП) |
| counterparty | VARCHAR(36) | GUID контрагента (WB / Ozon / YM / опт) |
| account_dt | VARCHAR(20) | Счёт дебет |
| account_ct | VARCHAR(20) | Счёт кредит |
| organization | VARCHAR(500) | Организация |
| amount_dt | DECIMAL(15,2) | Сумма оборота Дт |
| amount_ct | DECIMAL(15,2) | Сумма оборота Кт |
| quantity_dt | DECIMAL(15,3) | Количество оборота Дт |

Расчёт: выручка = `amount_ct WHERE account_ct = '90.01.1'`; себестоимость = `amount_dt WHERE account_dt = '90.02.1'`; маржа = выручка − себестоимость.
